### 技術決策分析

**為什麼選擇 Yelp Fusion API？**
1. **穩定性**：官方 API，不會被阻擋或封鎖
2. **效率**：結構化 JSON 資料，易於處理和去重
3. **成本**：免費額度足夠需求（500 次/天 vs 8 次搜尋）
4. **品質**：直接從 Yelp 資料庫獲取，資料準確可靠
5. **合規性**：符合 Yelp 使用條款，避免法律風險

**與直接爬取的比較：**
- ❌ 直接爬取：403 錯誤、反偵測複雜、不穩定
- ✅ API 方案：穩定、高效、合規、可擴展

### 當前阻塞問題

**hCaptcha 重複驗證問題：**
- **現象**：完成 hCaptcha 驗證後重複跳出，無法完成 Yelp 帳號註冊
- **影響**：無法申請 API Key，阻礙爬蟲開發進度
- **解決方案**：清除瀏覽器資料、無痕模式、更換瀏覽器、VPN 更換 IP
- **明日優先處理**：按順序嘗試各種解決方案

## 💡 關鍵學習與洞察

### **從台北到費城的策略演進**

**台北模式：**
- 抽獎激勵用戶推薦餐廳
- 物質獎勵驅動（酒商禮券、酒杯）
- 一次性參與為主

**費城模式升級：**
- 榮譽系統取代物質抽獎
- 專業認同和社群歸屬驅動
- 長期持續參與機制
- 創始成員特殊身份
- 更低成本、更可持續

**為什麼這樣調整？**
1. **成本考量**：海外專案初期無法負擔持續的物質獎勵
2. **文化差異**：美國用戶更重視專業認同和社群地位
3. **可擴展性**：榮譽系統可以無限擴展，物質獎勵不行
4. **長期價值**：建立真正的專家社群而非獎品獵人

### **兩個專案的互補價值**

* **台北專案**：驗證核心技術和產品概念
* **費城專案**：驗證商業模式和國際擴展性
* **互相學習**：費城的榮譽系統經驗可回饋到台北

## 🚨 當前技術挑戰與解決方案

### **Yelp 爬蟲技術演進**

**第一版嘗試（直接爬取）：**
- 使用 requests + BeautifulSoup
- 遇到 403 Forbidden 錯誤
- 原因：Yelp 反爬蟲機制（User-Agent 偵測、請求頻率限制）

**第二版嘗試（手動清單）：**
- 建立 15 家知名 BYOB 餐廳清單
- 問題：數量不足、不符合自動化需求
- 決定：刪除手動方案，專注自動化

**第三版方案（Yelp Fusion API）：**
- 使用官方 API，穩定可靠
- 每日 500 次免費請求，足夠需求
- 結構化 JSON 資料，易於處理
- 符合使用條款，避免法律風險

### **技術決策學習**

**從失敗中學習：**
1. **直接爬取風險高**：現代網站都有強力的反爬蟲機制
2. **手動方案不可擴展**：不符合大規模資料收集需求
3. **API 方案最穩定**：官方支援，長期可持續

**最佳實踐：**
- 優先考慮官方 API
- 準備多種備用方案
- 重視合規性和可持續性
- 快速迭代和調整策略
