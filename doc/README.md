# 🍷 BYOB 專案總覽文檔

## 📋 專案概述

BYOB (Bring Your Own Bottle) 是一個自帶酒水餐廳推薦平台，目前運營兩個獨立專案：

### 🇹🇼 台北 BYOB（主要專案）
- **定位**：台北自帶酒水餐廳推薦平台
- **階段**：核心系統完成，推廣與酒商合作階段
- **策略**：抽獎激勵 + 多平台推廣

### 🇺🇸 費城 BYOB（新專案）
- **定位**：Yelp 的 BYOB 專業補充平台
- **階段**：重複餐廳處理機制完成，準備網站英文化與留言機制討論
- **策略**：多平台資料收集 + 雙向表單驗證 + 榮譽系統驅動 + 餐廳聯絡機制

## 🚀 最新進度（2025年11月5日）

### 今日完成：重複餐廳處理機制實作完成

**🎯 關鍵成就**

* ✅ **重複餐廳標題自動加註**
  * 當系統檢測到重複餐廳時，自動在標題後方加註「(重複)」
  * 例如：「ABC餐廳」→「ABC餐廳(重複)」
  * 即使多次重複也只加一次（避免重複加註）

* ✅ **推薦次數欄位新增**
  * 新增 `recommendation_count` ACF 欄位，記錄餐廳被推薦的次數
  * 新建餐廳時預設為 1，可在 WordPress 後台手動修改
  * 追蹤餐廳受歡迎程度

* ✅ **重複檢查邏輯優化**
  * 修改 `byob_check_duplicate_restaurant()` 函數，新增專案類型篩選
  * 費城專案只檢查費城餐廳（透過 `source` 欄位篩選）
  * 避免與台北專案餐廳誤判為重複

* ✅ **相似度計算優化**
  * **地址為空時的處理**：即使地址為空也能正確比對重複餐廳
  * **英文標點符號處理**：統一處理逗號、句號、破折號、連字符等
  * **地址縮寫統一**：Street → St、Avenue → Ave、Philadelphia → Philly 等
  * 使用字邊界（`\b`）確保只匹配完整單字

* ✅ **除錯日誌新增**
  * 記錄重複檢查查詢結果和相似度計算過程
  * 方便後續問題診斷和系統優化

**🔧 技術架構**
- 專案類型篩選機制：避免跨專案誤判，提高準確性
- 智能相似度計算：處理各種邊緣情況（地址為空、格式差異等）

- 標題自動標記：方便管理員識別重複餐廳

## 💡 關鍵策略差異

### 台北模式 vs 費城模式

**台北模式：**
- 抽獎激勵、物質獎勵驅動、一次性參與

**費城模式升級：**
- 榮譽系統、專業認同驅動、長期持續參與
- 創始成員特殊身份、更低成本、更可持續

**調整原因：**
1. **成本考量**：海外專案初期無法負擔持續物質獎勵
2. **文化差異**：美國用戶更重視專業認同和社群地位
3. **可擴展性**：榮譽系統可無限擴展，物質獎勵不行
4. **長期價值**：建立專家社群而非獎品獵人

### 餐廳聯絡與驗證 ⚠️
- **挑戰**：餐廳 email 難以取得、餐廳老闆回覆率可能較低

### 資料衝突處理 ⚠️
- **挑戰**：多名網友提供不同資訊，難以判斷正確性
- **進展**：✅ 重複餐廳處理機制已完成（11/5），包含標題加註和推薦次數追蹤
- **下一步**：建立衝突檢測與解決機制、優先採用餐廳老闆驗證、保留多版本供人工審核

- **挑戰**：新帳號可能被視為推廣或 spam
- **解決方案**：先建立信譽，提供有價值的建議，使用追蹤系統

### 資料品質控制
- **挑戰**：Reddit 回覆可能包含錯誤資訊
- **解決方案**：交叉驗證、記錄來源、保留原始資料備份
