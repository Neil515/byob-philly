## 📝 技術筆記

### 餐廳業者註冊通知系統的技術要點

**通知機制設計：**
1. **Hook觸發**：使用WordPress `user_register` Hook自動觸發
2. **延遲執行**：使用 `wp_schedule_single_event()` 延遲5秒執行
3. **手動觸發**：提供 `byob_trigger_member_notification()` 函數供直接加入流程使用
4. **智能識別**：自動判斷註冊方式（邀請碼註冊/直接加入）

**餐廳名稱獲取邏輯：**
1. **優先獲取**：從 `_owned_restaurant_id` meta獲取餐廳ID
2. **備用機制**：從用戶 `display_name` 提取餐廳名稱
3. **除錯記錄**：完整的 `error_log()` 記錄方便追蹤問題
4. **關聯確保**：確保用戶和餐廳關聯正確建立後再發送通知

**Email通知內容：**
1. **完整資訊**：餐廳名稱、聯絡人、Email、註冊時間、註冊方式
2. **管理連結**：直接連結到後台管理頁面
3. **專業格式**：HTML格式設計，符合品牌形象
4. **安全處理**：使用 `esc_html()` 防止XSS攻擊

### Google Places API網格搜尋系統的技術要點

**網格搜尋（Grid Search）策略：**
1. **突破API限制**：Google Places API Text Search最多只能返回60筆結果
2. **多中心點搜尋**：使用12個台北主要區域作為搜尋中心點
3. **網格點生成**：每個中心點生成11×11=121個網格點
4. **搜尋記錄**：使用JSON檔案記錄已搜尋的網格點，避免重複搜尋
5. **斷點續傳**：支援中斷後從上次位置繼續搜尋

**網格搜尋設定參數：**
1. **網格大小**：0.01度（約1.1公里）
2. **搜尋半徑**：0.03度（約3公里）
3. **中心點數量**：12個台北主要區域
4. **預期結果**：每個區域最多7,260筆（121×60），去重複後約1,000-3,000筆

**搜尋記錄機制：**
1. **記錄檔案**：search_log.json
2. **識別鍵值**：query_key = f"{query}_{GRID_SIZE}_{GRID_RADIUS}"
3. **自動過濾**：載入已完成的網格點，只搜尋剩餘的點
4. **進度保存**：每10個網格點自動保存進度
