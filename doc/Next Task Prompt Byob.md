## 🗓️ 明日（2025-11-02）工作規劃

### 🎯 核心任務：多層次驗證回覆率處理策略

**問題背景**：
Crowdsourced 專案中最關鍵的風險是**驗證回覆率過低**，導致：
- 資料品質失衡
- 信心度不足
- 影響整體可信度與用戶信任
- 大量餐廳資料停滯在「待驗證」狀態

**目標**：建立一套完整的多層次處理策略，確保即使餐廳老闆不回覆，也能維持平台資料品質和用戶體驗。

---

### 📋 策略 1：自動化信心度動態調整

#### 1.1 建立信心度下降機制
- **目的**：隨著時間推移，未驗證資料的信任度自然降低
- **規則設計**：
  ```
  初始信心度：High/Medium/Low（基於多個來源）
  
  信心度下降時間表：
  - 1-3個月：維持原信心度
  - 3-6個月：下降一級（High→Medium, Medium→Low）
  - 6個月以上：下降二級（High→Low, Medium→Low）
  - Low 維持不變
  ```

- **實作方式**：
  - ACF 新增欄位：`last_verified_date`, `confidence_decay_tier`
  - WordPress Cron Job 每日檢查並更新信心度
  - 前台顯示：根據最新信心度顯示不同視覺標記

#### 1.2 多重來源交叉驗證
- **目的**：即使餐廳不回應，也能透過社群回饋提高可信度
- **策略**：
  - Reddit 貼文自動定期更新（每 3 個月一次）
  - 收集網友實際用餐回饋
  - 記錄驗證次數和來源多樣性
  - 當驗證來源 ≥ 3 個不同渠道時，自動提升信心度

---

### 📋 策略 2：分層顯示與用戶教育

#### 2.1 餐廳狀態分層顯示
- **目的**：讓用戶清楚知道資料可靠性
- **顯示設計**：
  ```
  高信心度 (High Confidence) - 綠色標籤
  - 餐廳老闆直接驗證
  - 或多來源交叉驗證（≥3個）
  - 最近更新 < 3個月
  
  中信心度 (Verified by Community) - 黃色標籤
  - 社群驗證但老闆未回應
  - 或老闆驗證但 > 3個月
  
  低信心度 (Unverified) - 灰色標籤
  - 僅單一來源
  - 或 > 6個月未更新
  - 顯示「需要更新」提醒
  ```

#### 2.2 透明度與用戶教育
- **「關於我們」頁面說明**：
  - 清楚解釋三級信心度系統
  - 說明資料驗證流程
  - 邀請用戶提供回饋和更新
- **餐廳頁面友好提醒**：
  - 低信心度：顯示「這筆資料需要更新，如果您最近造訪過，請提供回饋」
  - 中信心度：顯示「資料基於社群回饋，歡迎實際體驗後驗證」

---

### 📋 策略 3：持續追蹤與自動化提醒

#### 3.1 智慧型追蹤系統
- **目的**：確保長期資料活力
- **追蹤機制**：
  - 記錄首次提交日期、最後驗證日期
  - 記錄聯絡嘗試次數和回應狀態
  - 追蹤社群回饋更新頻率
  - 記錄信心度變化歷史

#### 3.2 分階段提醒策略
- **第一階段（1-3個月）**：
  - 每 2 週自動發送一次確認邀請
  - 使用溫和、友善的語氣
  - 強調共同建設費城 BYOB 社群

- **第二階段（3-6個月）**：
  - 降為每月 1 次提醒
  - 加入社群回饋統計（「已有 X 位用戶造訪」）
  - 提供更彈性的回應選項

- **第三階段（6個月以上）**：
  - 停發給老闆的提醒
  - 改為每季發送社群調查貼文
  - 允許匿名回饋

---

### 📋 策略 4：內容品質控制

#### 4.1 資訊完整度評分
- **目的**：即使未驗證，也要確保基本資訊準確
- **評分標準**：
  ```
  高品質（8-10分）：姓名、地址、電話、網站、開瓶費政策、設備、服務等級、類型、備註
  中品質（5-7分）：至少 6 個核心欄位
  低品質（<5分）：< 6 個核心欄位
  ```

- **前台顯示邏輯**：
  - 高品質 + 中高信心度 → 正常顯示
  - 中品質 → 顯示「資訊不完整」
  - 低品質 → 顯示「資訊不足」或暫時隱藏

#### 4.2 社群輸入驗證機制
- **Reddit 回覆驗證**：
  - 優先採用多個不同用戶的相似回覆
  - 識別可疑或偏頗的資訊
  - 對矛盾資訊進行標記

---

### 📋 策略 5：彈性資料管理

#### 5.1 隱藏 vs 標記
- **隱藏條件**：
  - 信心度 = Low 且品質 < 5 分且 > 1年未更新
  - 餐廳確認已歇業
- **標記顯示**：
  - 其他所有狀況都顯示，但加上適當標籤
  - 讓用戶有資訊可參考，同時了解限制

#### 5.2 資料檔案管理
- **維護「歷史資料」檔案**：
  - 儲存所有未驗證但可能有用的資訊
  - 作為未來重新激活的基礎
  - 用於分析驗證回覆率趨勢

---

### 📋 策略 6：社群驅動驗證

#### 6.1 榮譽系統整合
- **驗證徽章獎勵**：
  - 「資料驗證者」徽章
  - 「BYOB Verifier」專家徽章
  - 激勵用戶定期更新和驗證餐廳資訊

#### 6.2 社群投票機制
- **「這資訊正確嗎？」按鈕**：
  - 用戶可標記資訊正確/錯誤/過時
  - 達到一定門檻（如 3 次錯誤標記）觸發重新驗證
  - 最活躍的驗證者獲得特殊識別

## 🧭 重要技術學習與踩雷紀錄

### 1) 單選題加上 placeholder 後 ACF 回退問題
- **根因**：WP 端把表單「顯示文字」直接寫入 ACF；ACF 期望的是「值鍵」（key）
- **解法**：在 WP `functions.php` 寫入安全映射（就地 if/elseif），將顯示文字 → 值鍵
  - `philly_corkage_fee`：Free → `free`、Corkage Fee → `corkage_fee`、Other → `other`、未選擇 → ''
  - `byob_service_level`：四個長句對應 `full_service`/`basic_service`/`self_service`/`no_service`、未選擇 → ''
  - `show_reddit_username`：以 Yes/No 前綴判斷，並規一撇號與空白；Yes → `yes`、No → `no`、未選擇 → ''

### 2) 餐廳類型/酒器設備的 other 與備註
- **根因**：Apps Script/ACF 欄位鍵不一致，以及將中文「其他」存入導致條件顯示不觸發
- **解法**：
  - ACF 勾選鍵一律使用英文 `'other'`
  - 若有說明文字，確保陣列包含 `'other'`，並寫入對應 other_note
  - 前台顯示：把字串中的 `'other'` 替換為 `Other: [note]`（`archive-restaurant.php` 和 `single_restaurant.php` 已處理）

### 3) API 參數硬編碼問題
- **現象**：`source` 欄位在 `functions.php` 被硬編碼為 `'philly_community_recommendation'`
- **根因**：未從 API 請求中讀取 `source` 參數
- **解法**：
  - 在 API 端點定義中新增 `source` 參數（190-193 行）
  - 從 `$request->get_param('source')` 讀取（761 行）
  - 移除所有硬編碼，改用動態值（784 行、980 行）
- **影響**：兩個表單現在可正確區分資料來源

## 🚨 當前挑戰與風險

### **驗證回覆率過低風險** ⚠️（明日重點）
- **風險**：餐廳老闆回覆率可能僅 20-30%，大量資料停滯待驗證
- **影響**：資料品質失衡、信心度不足、用戶信任下降
- **緩解策略**：多層次處理系統（見明日工作規劃詳述）

### **Reddit 社群接受度**
- **風險**：新帳號可能被視為推廣或 spam
- **緩解**：先建立信譽，提供有價值的建議
- **備案**：準備多個社群平台互動

### **資料品質控制**
- **風險**：Reddit 回覆可能包含錯誤資訊
- **緩解**：交叉驗證多個回覆，記錄資訊來源
- **備案**：保留原始爬蟲資料作為備份

### **內容創作挑戰**
- **風險**：缺乏實際用餐經驗
- **緩解**：基於爬蟲資料和社群回饋
- **備案**：標明資料來源，邀請用戶補充
